# Requirements Document

## Introduction

This document specifies the requirements for integrating `clang-tidy` static
analysis into the Kythira build system and pre-commit hook. The goal is to
catch a defined class of bugs, anti-patterns, and style violations
automatically, building on the formatting enforcement already provided by
`clang-format`.

`clang-tidy` is substantially slower than `clang-format`: a single
template-heavy C++23 translation unit in this codebase can take 10–60 seconds.
The integration therefore differs from the format check in two ways: the
pre-commit check is **opt-in by default** (enabled by setting
`TIDY_CHECK=1`), and the CMake target supports parallel execution to make
project-wide analysis practical. Developers are expected to run
`TIDY_CHECK=1 git commit` — or the CMake target — before opening a pull
request or on branches where correctness concerns warrant a deeper check.

## Glossary

- **Tidy_Config**: The `.clang-tidy` file at the repository root that defines
  the enabled checks, per-check options, and project-wide suppressions.
- **Compilation_Database**: The `compile_commands.json` file generated by
  CMake in the build directory (`build/`). `clang-tidy` uses it to obtain
  include paths and compiler flags for each translation unit.
- **Static_Analysis_Target**: The CMake `static-analysis` target that runs
  `clang-tidy` on all project `.cpp` sources using the Compilation_Database.
- **Static_Analysis_Fix_Target**: The CMake `static-analysis-fix` target that
  runs `clang-tidy --fix` to auto-apply machine-fixable suggestions.
- **Pre_Commit_Tidy**: The optional tidy step in the pre-commit hook that
  runs `clang-tidy` on staged `.cpp` files only. Enabled by setting
  `TIDY_CHECK=1` at commit time.
- **HeaderFilterRegex**: The `.clang-tidy` pattern that restricts diagnostic
  output to project-owned headers, excluding third-party and system headers.
- **NOLINT**: An inline suppression comment (`// NOLINT(check-name)` or
  `// NOLINT`) that silences a specific diagnostic on a given line.

## Requirements

### Requirement 1

**User Story:** As a developer, I want a single canonical `.clang-tidy`
configuration at the repository root so that all contributors and tools use
the same set of checks and suppressions automatically.

#### Acceptance Criteria

1. WHEN the repository is cloned THEN a `.clang-tidy` file SHALL exist at the
   repository root.
2. WHEN `clang-tidy` is invoked from any subdirectory THEN it SHALL find the
   root `.clang-tidy` via the standard parent-directory search.
3. WHEN the `.clang-tidy` file is parsed by `clang-tidy --dump-config` THEN it
   SHALL produce no errors or unrecognised-option warnings.
4. WHEN the enabled checks are defined THEN they SHALL be selected from the
   following groups (with specific sub-checks disabled where they are too noisy
   or conflict with project patterns):
   - `bugprone-*`: catches real latent bugs
   - `modernize-*`: enforces C++23-appropriate idioms
   - `readability-*`: enforces consistent naming and structural conventions
   - `performance-*`: catches unnecessary copies and move-fixable patterns
   - `cppcoreguidelines-*`: C++ Core Guidelines violations
   - `clang-analyzer-*`: deep static analysis paths
5. WHEN a check is disabled THEN the `.clang-tidy` file SHALL include a comment
   explaining why, so suppressions are auditable.
6. WHEN `HeaderFilterRegex` is set THEN it SHALL include only project-owned
   headers (`include/` and `tests/`) and exclude vcpkg, Folly, Boost, and
   system paths, so findings in third-party code are not reported.

### Requirement 2

**User Story:** As a developer, I want a CMake `static-analysis` target that
runs `clang-tidy` over all project sources so that I can audit the full tree
with one command before opening a pull request.

#### Acceptance Criteria

1. WHEN the `static-analysis` target is invoked THEN it SHALL run `clang-tidy`
   on every `.cpp` source file under `src/`, `tests/`, and `examples/` using
   the Compilation_Database from `build/`.
2. WHEN `run-clang-tidy` is available on `PATH` or in `~/.local/bin` THEN the
   target SHALL use it for parallel execution (`-j $(nproc)`) to reduce
   wall-clock time.
3. WHEN `run-clang-tidy` is not available THEN the target SHALL fall back to
   running `clang-tidy` on each source file sequentially using a shell loop.
4. WHEN the Compilation_Database (`build/compile_commands.json`) is absent
   THEN the target SHALL fail with a clear message instructing the developer to
   run `cmake -S . -B build` first.
5. WHEN `clang-tidy` is not found THEN CMake SHALL emit a warning at configure
   time and define a stub `static-analysis` target that prints an actionable
   error rather than silently succeeding or failing cryptically.
6. WHEN the target completes with zero findings THEN it SHALL print a
   one-line confirmation.
7. WHEN the target reports findings THEN it SHALL exit with a non-zero status
   so that CI can detect the failure.

### Requirement 3

**User Story:** As a developer, I want a CMake `static-analysis-fix` target
that auto-applies machine-fixable suggestions so that I can correct entire
check categories in one step.

#### Acceptance Criteria

1. WHEN the `static-analysis-fix` target is invoked THEN it SHALL run
   `clang-tidy --fix` on all project `.cpp` sources.
2. WHEN `--fix` is used THEN only checks with safe auto-fixers SHALL be applied;
   the `.clang-tidy` configuration SHALL not enable destructive fixers
   (e.g. `modernize-use-default-member-init` with non-trivial constructors).
3. WHEN `static-analysis-fix` is invoked THEN it SHALL be subject to the same
   tool-availability guard as `static-analysis` (Req 2.5).

### Requirement 4

**User Story:** As a developer, I want the pre-commit hook to optionally run
`clang-tidy` on staged `.cpp` files so that I can catch issues before they
land on the main branch, without being blocked on every WIP commit.

#### Acceptance Criteria

1. WHEN `TIDY_CHECK=1` is NOT set THEN the pre-commit tidy step SHALL be
   silently skipped, and a one-line notice SHALL be printed indicating how to
   enable it (e.g., `[tidy] Skipped (set TIDY_CHECK=1 to enable)`).
2. WHEN `TIDY_CHECK=1` is set THEN the hook SHALL identify staged `.cpp` files
   via `git diff --cached --name-only --diff-filter=ACMR`; `.hpp` files SHALL
   be excluded because headers are analysed transitively when their
   corresponding translation units are processed, and running tidy on headers
   directly produces many context-free false positives.
3. WHEN no staged `.cpp` files are present THEN the tidy step SHALL be skipped
   silently even when `TIDY_CHECK=1` is set.
4. WHEN the Compilation_Database (`build/compile_commands.json`) is absent AND
   `TIDY_CHECK=1` is set THEN the hook SHALL print a clear message and skip the
   tidy step rather than auto-configuring the build directory (unlike the
   coverage check, which does auto-configure, because a tidy-only configure
   would leave the user with an unchecked debug build).
5. WHEN `clang-tidy` is not found on `PATH` or `~/.local/bin` THEN the hook
   SHALL print a warning and skip gracefully.
6. WHEN one or more staged `.cpp` files produce tidy findings THEN the hook
   SHALL print the findings, instruct the developer to either fix them or add
   per-line `// NOLINT(check-name)` suppressions, and abort the commit with
   exit status 1.
7. WHEN `SKIP_TIDY_CHECK=1` is set THEN the hook SHALL skip the tidy step
   entirely and print `[tidy] Skipped (SKIP_TIDY_CHECK=1)`.

### Requirement 5

**User Story:** As a developer, I want the pre-commit tidy check to run after
the format check and before the coverage ratchet so that cheap checks precede
expensive ones.

#### Acceptance Criteria

1. WHEN the pre-commit hook runs THEN the step order SHALL be:
   1. `clang-format` check (fast, always active)
   2. `clang-tidy` check (slow, opt-in)
   3. coverage ratchet (slow, always active)
2. WHEN the format check fails THEN the tidy and coverage steps SHALL NOT run.
3. WHEN the tidy check fails THEN the coverage step SHALL NOT run.

### Requirement 6

**User Story:** As a developer, I want a documented suppression mechanism so
that I can silence false positives without disabling an entire check category.

#### Acceptance Criteria

1. WHEN a specific line produces a false positive THEN the developer MAY add
   `// NOLINT(check-name)` at the end of that line to suppress exactly that
   check on that line.
2. WHEN a suppression is added THEN the check name SHALL be explicit (e.g.,
   `// NOLINT(bugprone-unchecked-optional-access)`) rather than a bare
   `// NOLINT`, so suppressions are searchable and auditable.
3. WHEN an entire source file warrants suppression (e.g., a generated file or
   a third-party shim) THEN a `// NOLINTBEGIN` / `// NOLINTEND` block SHOULD
   be used rather than per-line annotations.
4. WHEN a check is globally noisy across the project THEN it SHALL be disabled
   in `.clang-tidy` with an explanatory comment rather than scattered
   `// NOLINT` annotations.

### Requirement 7

**User Story:** As a project maintainer, I want the clang-tidy integration
documented so that contributors understand how to run the analysis, interpret
findings, and add suppressions.

#### Acceptance Criteria

1. WHEN the spec is implemented THEN `README.md` SHALL include a
   "Static Analysis" section explaining: the CMake targets, how to run the
   pre-commit tidy check, and how to suppress false positives.
2. WHEN the "Static Analysis" section is written THEN it SHALL document
   `TIDY_CHECK=1` and `SKIP_TIDY_CHECK=1` alongside the existing
   `SKIP_FORMAT_CHECK=1` and `SKIP_COVERAGE_CHECK=1` documentation.
3. WHEN the spec is complete THEN `doc/TODO.md` SHALL mark the
   `clang-tidy integration` item done and add it to the "What Changed" summary.
